Add rayon parallelism to the three hot loops in `run_elkan()`. The `parallel` feature and rayon are already available.

**1. Parallelize initial assignment (lines 390-406):**

Replace the sequential `for i in 0..n_samples` loop with a parallel iteration. Since `labels`, `upper`, and `lower` need per-element writes, use index-based parallel chunks:

```rust
use rayon::prelude::*;

// Split work into chunks for parallel initial assignment
let chunk_results: Vec<(i32, f64, Vec<f64>)> = (0..n_samples)
    .into_par_iter()
    .map(|i| {
        let xi = x_row(i);
        let mut min_dist = f64::MAX;
        let mut min_idx = 0usize;
        let mut lowers = vec![0.0f64; k];
        for j in 0..k {
            let cj = centers.row(j);
            let cj_s = cj.as_slice().expect("SAFETY: standard-layout array");
            let dist = crate::linalg::squared_euclidean_distance(xi, cj_s).sqrt();
            lowers[j] = dist;
            if dist < min_dist {
                min_dist = dist;
                min_idx = j;
            }
        }
        (min_idx as i32, min_dist, lowers)
    })
    .collect();

for (i, (label, upper_bound, lowers)) in chunk_results.into_iter().enumerate() {
    labels[i] = label;
    upper[i] = upper_bound;
    for j in 0..k {
        lower[i * k + j] = lowers[j];
    }
}
```

However, this allocates a Vec<f64> per sample. A better approach uses `par_chunks_mut` on the output arrays. The cleanest approach for Elkan:

Use `rayon::scope` or parallel iterators with mutable slice chunks. Since `labels`, `upper`, and `lower` are independent per sample, split them into parallel chunks:

```rust
#[cfg(feature = "parallel")]
{
    let labels_slice = labels.as_slice_mut().expect("SAFETY: contiguous");
    let chunk_size = (n_samples / rayon::current_num_threads()).max(64);

    labels_slice.par_chunks_mut(chunk_size)
        .zip(upper.par_chunks_mut(chunk_size))
        .zip(lower.par_chunks_mut(chunk_size * k))
        .enumerate()
        .for_each(|(chunk_idx, ((labels_chunk, upper_chunk), lower_chunk))| {
            let start = chunk_idx * chunk_size;
            for (local_i, ((label, ub), lower_row)) in labels_chunk.iter_mut()
                .zip(upper_chunk.iter_mut())
                .zip(lower_chunk.chunks_exact_mut(k))
                .enumerate()
            {
                let i = start + local_i;
                let xi = &x_data[i * n_features..(i + 1) * n_features];
                // ... same logic as sequential
            }
        });
}
```

Actually, the simplest correct approach: keep the x_row closure but use indices. Since labels/upper/lower don't overlap across samples, we can use unsafe split-at-mut or simpler approaches. The CLEANEST approach:

**Use a parallel collect + scatter pattern for initial assignment and Step 2.**

For Step 2 (the main hot loop, lines 439-492), this is more complex because it modifies `labels[i]`, `upper[i]`, and `lower[i*k..i*k+k]` — all per-sample, no cross-sample dependencies. Use the same parallel chunk pattern.

**For Step 3 (center update, lines 494-514):** This requires a reduction (accumulate per-cluster). Use rayon `par_iter` with `fold` + `reduce`:

```rust
#[cfg(feature = "parallel")]
{
    let (par_centers, par_counts) = (0..n_samples)
        .into_par_iter()
        .fold(
            || (Array2::zeros((k, n_features)), vec![0usize; k]),
            |(mut acc_centers, mut acc_counts), i| {
                let ci = labels[i] as usize;
                acc_centers.row_mut(ci).add_assign(&x.row(i));
                acc_counts[ci] += 1;
                (acc_centers, acc_counts)
            },
        )
        .reduce(
            || (Array2::zeros((k, n_features)), vec![0usize; k]),
            |(mut a_c, mut a_n), (b_c, b_n)| {
                a_c += &b_c;
                for j in 0..k { a_n[j] += b_n[j]; }
                (a_c, a_n)
            },
        );
    new_centers.assign(&par_centers);
    counts.copy_from_slice(&par_counts);
}
```

**For Step 5 (bound update):** This is per-sample independent — use par_iter.

**Important constraints:**
- Gate all parallel code behind `#[cfg(feature = "parallel")]` with sequential fallback
- The `x_row` closure borrows x_data — need to use x_data directly in parallel code (closures can't be shared across threads unless they're Fn+Send+Sync)
- Keep the sequential path as fallback for `#[cfg(not(feature = "parallel"))]`
- Do NOT change the algorithm semantics — same Elkan triangle inequality logic
- Run `cargo fmt --all` after changes

Minimum viable parallelism: parallelize Step 2 (label reassignment) and Step 3 (center update). These are the biggest hot paths. Initial assignment runs once, Step 5 is cheaper. Focus effort on Step 2 + Step 3.
