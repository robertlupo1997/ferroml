Add optional GPU acceleration for compute-heavy operations (MLP GEMM, KMeans distance matrix) via wgpu, and expand benchmark coverage to include all missing models plus GPU vs CPU comparisons.

## Phase 8.4: GPU Acceleration

### Phase 8.4.1: GPU Module Structure & GpuBackend Trait

**Overview**: Create the GPU module with a backend abstraction trait, wgpu device initialization, and feature gating.

**Changes Required**:

1. **File**: `ferroml-core/Cargo.toml`
   - Add feature: `gpu = ["wgpu", "bytemuck"]`
   - Add dependencies:
     ```toml
     wgpu = { version = "24", optional = true }
     bytemuck = { version = "1", features = ["derive"], optional = true }
     ```

2. **File**: `ferroml-core/src/gpu/mod.rs` — CREATE
   - `GpuBackend` trait:
     ```rust
     pub trait GpuBackend: Send + Sync {
         fn matmul(&self, a: &Array2<f64>, b: &Array2<f64>) -> Result<Array2<f64>>;
         fn pairwise_distances(&self, x: &Array2<f64>, centers: &Array2<f64>) -> Result<Array2<f64>>;
         fn is_available(&self) -> bool;
     }
     ```
   - `GpuDevice` struct wrapping `wgpu::Device` + `wgpu::Queue`
   - `GpuDevice::new() -> Result<Self>` — async adapter request, device creation
   - `GpuDevice::try_new() -> Option<Self>` — returns None if no GPU available
   - Feature-gated: entire module behind `#[cfg(feature = "gpu")]`

3. **File**: `ferroml-core/src/gpu/kernels.rs` — CREATE
   - WGSL compute shaders as `const &str`:
     - `MATMUL_SHADER`: tiled matrix multiply (f32, 16x16 tiles)
     - `DISTANCE_SHADER`: pairwise squared Euclidean distance
   - Note: wgpu only supports f32 in shaders. We cast f64→f32 on upload, f32→f64 on download. Document precision implications.

4. **File**: `ferroml-core/src/gpu/backend.rs` — CREATE
   - `WgpuBackend` implementing `GpuBackend`
   - Buffer management: create/upload/download GPU buffers
   - Pipeline caching: store compiled compute pipelines
   - `matmul` implementation: upload A,B → dispatch shader → download C
   - `pairwise_distances` implementation: upload X, centers → dispatch → download distance matrix

5. **File**: `ferroml-core/src/lib.rs`
   - Add `#[cfg(feature = "gpu")] pub mod gpu;`

**Success Criteria**:
- [ ] Automated: `cargo check -p ferroml-core --features gpu` compiles
- [ ] Automated: `cargo test -p ferroml-core --lib --features gpu -- gpu` passes basic matmul + distance tests
- [ ] Manual: Verify GPU device initializes on machine with GPU

**Risks**:
- wgpu may not find a GPU adapter in CI (headless). Mitigation: all GPU tests skip gracefully if no adapter found.
- f64→f32 precision loss. Mitigation: document tolerance, use f64 CPU fallback for high-precision needs.

## Risks & Mitigations

1. **No GPU in CI**: wgpu may not find adapter in GitHub Actions. Mitigation: all GPU tests use `#[ignore]` or runtime skip. CI runs without `gpu` feature.
2. **f32 precision**: wgpu compute shaders only support f32. Mitigation: document tolerance, GPU is optional enhancement not replacement.
3. **wgpu async initialization**: Device creation is async. Mitigation: use `pollster::block_on()` for sync wrapper.
4. **Build time**: wgpu is a large dependency. Mitigation: strictly optional via feature flag.
5. **Cross-platform**: wgpu supports Vulkan/Metal/DX12 but behavior varies. Mitigation: test on available hardware, document requirements.
